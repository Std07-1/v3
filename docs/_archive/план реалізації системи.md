# план реалізації системи

Згідно з документом від **2026-02-28** (ADR-0024), “технічно досконала система без правильної презентації = ще один індикатор”, а ціль — **AI-помічник, який думає як smart money**; незамінність для сильного трейдера будується не кількістю оверлеїв, а **ранжуванням (POI), наративом (why→action) і (далі) replay-тренажером**.

Нижче — план “як саме подавати” зони/структуру/рівні/пули/магніти/Premium-Discount так, щоб v3 SMC Terminal відповідав **меті v1** і був читабельний “за 3 секунди”, без кладовища на чарті.

---

## 0) Конституція подачі (інваріанти UI/SMC)

Це не “побажання”, а правила, які визначають дизайн і фільтри відображення:

1. **Правило 3 секунд:** на відкритті графіка трейдер має миттєво побачити **bias, POI, target, invalidation**.
2. **Object budget у Focus:** ≤ **12** SMC-об’єктів на чарті завжди (з капами по сторонам/типам).
3. **Zero-noise поза контекстом:** якщо ціна далеко від POI — чарт майже чистий (свічки + мінімальні підказки).
4. **Ієрархія:** HTF context → Structure POI → Execution trigger (ніколи навпаки). Це закладено і як tf-roles (HTF/Structure/Execution).
5. **“WAIT / Нічого не робимо” — нормальний стан**, не помилка: банер WAIT/CONFLICTING + чистий чарт — очікувана поведінка.
6. **Decay/TTL проти “кладовища”**: старі/мітиговані елементи мають зникати з Focus за TTL, а не жити вічно.
7. **Config-driven візуальна мова:** кольори/opacity/TTL/бюджети — тільки через `config.json:smc.display`, без hardcode.

## 1) Інформаційна архітектура: 3 “поверхні” продукту

Щоб сильний трейдер не “скролив 13 графіків”, UI має бути розкладений на три поверхні:

### 1.1 Dashboard (пріоритизація символів)

Трейдер починає не з чарта, а з **таблиці пріоритетів**: де зараз **найкращі POI** (grade/score/bias). Це прямо описано як робочий сценарій “London open → Dashboard → фокус на 1 символі”.

**Поля, які мають бути видимі без кліків:**

* `HTF bias` (bull/bear/neutral)
* `Top POI grade` (A+/A/…)
* `Distance to POI` (в ATR або пунктах)
* `Nearest liquidity target` (куди “тягне”)
* `State`: ACTIVE / WAIT / CONFLICTING

> Це і є “незамінність”: не просто намалювати зони, а **вирішити проблему вибору фокуса**.

### 1.2 Chart (три режими: Focus / Research / Debug)

За замовчуванням **Focus = production** (не “демо-mode”). Заборона показувати все одразу — явна політика ролі.

* **Focus:** максимум сигналу, мінімум шуму, об’єктний бюджет, один активний сценарій або WAIT.
* **Research:** розкриває “чому” (більше зон/рівнів), але з капами.
* **Debug:** безліміт, з технічними метаданими (для тебе/QA, не для трейдера).

### 1.3 Inspector/SMC Panel (наратив + trade-setup)

Це “ліва панель” або drawer, яка відповідає на 3 питання:

* **WHY** (фактори confluence)
* **WHAT** (action: buy/sell/wait)
* **HOW** (entry/SL/TP, R:R) — “one-click trade setup” / risk calculator як конкурентна перевага мети v1.

## 2) Таксономія SMC-об’єктів: що ми показуємо трейдеру

Щоб не змішувати поняття, фіксуємо “шари” (в цьому порядку):

### 2.1 Context layer (HTF)

* HTF bias + ключові HTF зони (OB/FVG) як **“HTF zone projection”** (не дублювати все на LTF).
* Premium/Discount — **фон**, не overlay (заборона V9).

### 2.2 POI layer (рішення/сценарій)

POI = **одна** “робоча” область, яку трейдер реально торгує.
У Focus одночасно **не більше 1 активного сценарію на символ** (або WAIT).

### 2.3 Execution layer (тригер)

На M5/M1 (execution tf) — **лише тригери**, а не весь “зоопарк” зон (це закладено в tf-role матриці).

### 2.4 Liquidity layer (цілі)

Тут важливо відрізнити:

* **Levels** (EQH/EQL, PDH/PDL, PWH/PWL…) — як тонкі горизонталі/мітки.
* **Pools** (кластери/зони ліквідності) — як напівпрозорі “смуги” (area).
* **Magnets** (вектор “тягне до…”) — як стрілка/маркер напрямку.
  У ваших контрактах viewer-стану це вже розведено як `pools` і `magnets`.
  У меті v1 “Liquidity Magnet” прямо визначений як диференціатор (візуальна стрілка/індикатор тяжіння).

## 3) Візуальна мова (Chart Language Spec): як це виглядає на екрані

### 3.1 Кольори та семантика (SSOT у config)

Базова палітра вже описана як конфіг (OB bull/bear, FVG bull/bear, premium/discount, BSL/SSL, PDH/PDL, CHoCH, inducement…).

Ключовий принцип: **A+ не може виглядати як C** (заборона V2).

### 3.2 Opacity/visibility = функція strength + grade + режим

У Focus слабке ховається, сильне — лишається; матриця “Strength → Visibility” і “Grade → Display priority” вже визначені.

### 3.3 Лейбли короткі (≤ 6 символів)

`OB`, `FVG`, `BOS ↘`, `CHoCH ↗`, `A+` — цього достатньо (заборона V8).

### 3.4 Premium/Discount

* Тільки якщо є валідний range;
* Малюється **фоном** (light tint), без “коробок” поверх свічок;
* У Focus — не обов’язково завжди, лише коли впливає на рішення (наприклад, POI у premium/discount).

## 4) Логіка відбору: як не перетворити чарт на смітник

Це “серце” подачі.

### 4.1 Фільтр proximity (контекстність)

Показуємо зони/цілі лише якщо ціна в “радіусі уваги”: `proximity_atr_mult` (типово 3×ATR) — інакше **чарт чистий** (AC-3).

### 4.2 Капи + детерміністичний ранк

На бекенді вже є жорсткі cap-и та `_zone_rank()` (kind priority → strength → freshness), і зменшені ліміти щоб прибрати clutter (N2/N3).

На UI-стороні це має бути **підсилено** “display-rank”:

* спочатку grade (A+…)
* потім proximity
* потім strength/свіжість (TTL/decay)

### 4.3 TTL/decay як закон

TTL по TF/grade (наприклад M15 B/C 100 bars, H4 A 500, D1 A 1000) визначено конфігом; mitigated зони теж мають короткий TTL у Focus.

## 5) Workflow Spec: як сильний трейдер реально працює (H4 → M15 → M5)

Це має бути відтворено 1:1 у UI, бо це “м’язова пам’ять” трейдера.

**Еталонний сценарій уже описаний у ADR-0024**:
Dashboard → вибір символу → H4 контекст/зона → M15 підтвердження CHoCH → entry, SL, TP по liquidity.

Важливо: у Focus система повинна вміти сказати “WAIT” і показати це як нормальний стан (а не “нічого не знайдено”).

---

## 6) Інтеракції: як трейдер “споживає” SMC без ментального шуму

### 6.1 Hover = короткий факт; Click = рішення

* Hover по POI/zone: `kind + grade + strength + age + next target`.
* Click: відкриває **Scenario Card** (в Inspector):

  * Action: BUY/SELL/WAIT
  * Entry/SL/TP (risk calculator)
  * “Why”: фактори confluence (OB+FVG+HTF+P/D+…)
  * “Invalidation”: що ламає сценарій

Заборона: narrative без action — неприпустимо.

### 6.2 One-click Risk Calculator (Should-have мети v1)

“Клік на OB → готовий сетап” — прямо в меті v1 як конкурентна перевага.

## 7) Що робимо далі в рамках ADR0024, щоб v1 реально став “незамінним”

Факт: SMC overlay pipeline вже реалізований (SmcRunner in-process → WS frames → UI v4 overlay), але E3-інтелект ще не зроблений (sessions/POI/narrative).

### P1 — “Display Doctrine” у прод-вигляді (закрити Focus/Research/Debug повністю)

* Уніфікувати всі відображення через `config.json:smc.display` (бюджети/TTL/кольори).
* Довести AC-1..AC-4 (3 секунди, бюджет, zero-noise, TTL).

### P2 — Sessions/Killzones (E3 prereq)

Це “Nice-to-have” мети v1, але для POI scoring потрібно як фактор якості/часу.

### P3 — Confluence POI + Grade A+/A/B/C

Саме ранжування POI — це “moat” рівня L2 (інтелект), який відсікає 80% шуму і робить продукт незамінним.

### P4 — Narrative Engine + Action language

Наратив у форматі “WHY → SO WHAT” (рекомендація дії) — це те, що відрізняє прямокутники від помічника.

### P5 — Dashboard + Alerts (Should-have мети v1)

* Dashboard “де зараз сетапи” (не скролити 13 графіків).
* Alerts “Bullish OB formed…” як окрема фіча (SMC-alerts).

---

## 8) Exit Gates (як перевіряємо, що подача «правильна»)

Мінімальний набір приймальних перевірок:

* **AC-1** 3-секундний тест на трейдерах (bias/POI/target/invalidation).
* **AC-2** автоматичний підрахунок visible objects ≤ 12 у Focus.
* **AC-3** zero-noise коли далеко від POI.
* **AC-4** TTL/anti-кладовище (B/C зони зникають з Focus після TTL).
* **WAIT-state**: система явно показує “нічого не робимо” (банер + чистий чарт).
