### ВІДКРИТІ ПИТАННЯ (потребують рішення)

#### ITEM-A: trade mode без вимоги ready trigger (§4, §20 item 5) — S2 policy
- ФАКТ: mode="trade" залежить від alignment+direction, НЕ від trigger state
- Тобто trade/approaching, trade/in_zone — валідні стани
- ПИТАННЯ: це свідомий product choice чи doctrinal mismatch?
- ВАРІАНТИ:
  A) Залишити як є: "trade" = "є сетап" (setup_mode), trigger = окремий індикатор readiness
  B) Перейменувати mode на "setup" щоб не плутати з "execute trade"
  C) Додати sub_mode "pending" для trade+approaching, "actionable" для trade+ready
- РИЗИК: Нульовий production ризик — це UI/UX семантика
- РЕКОМЕНДАЦІЯ: Варіант A (залишити). "trade" означає "сетап існує", trigger каже "наскільки готовий". Це вже працює і зрозуміло з headline.
- VERDICT: S3 COSMETIC. Не потребує код-змін. Можна уточнити в ADR як "design intent".

#### ITEM-B: lifecycle close-based vs narrative price-based invalidation (§8, §20 item 3) — S1 semantic
- ФАКТ коду:
  - engine.py: _update_zone_lifecycle() → mitigated if last_bar.c < zone.low (CLOSE)
  - narrative.py: _is_invalidated() → True if current_price < zone.low (LIVE PRICE)
- НАСЛІДОК: Narrative може показати "invalidated"/"wait" поки lifecycle ще тримає zone "active"
  Або навпаки: price wicked через zone → narrative invalidated → bar closed inside zone → lifecycle ще active
- ВАРІАНТИ:
  A) Narrative теж по close: `_is_invalidated()` бере last_bar.c замість current_price
     + Консистентно з lifecycle. - Narrative не реагує на intra-bar wick invalidation (stale scenario до нового бару)
  B) Залишити як є: narrative = "real-time caution", lifecycle = "confirmed state"
     + Narrative швидше реагує. - Семантична розбіжність
  C) Hybrid: narrative invalidation = (close-based OR price beyond zone + margin)
     + Best of both. - Складніше
- РИЗИК: UI може показувати "wait" для зони що lifecycle вважає active — шум для трейдера
- РЕКОМЕНДАЦІЯ: Варіант B (залишити). Narrative — advisory layer, не trading state machine.
  Price-based invalidation = "стоп, подивись уважніше". Lifetime bar.close = фінальне рішення.
  Це РІЗНІ рівні абстракції, не помилка. Але ПОТРІБНО задокументувати розбіжність в ADR-0033.
- VERDICT: S2. Задокументувати як design intent в ADR. Не потребує код-змін.

#### ITEM-C: _find_qualifying_choch() приймає BOS (§5.4) — S3 semantics
- ФАКТ: Функція named "_find_qualifying_choch" але фільтрує ("choch_X", "bos_X")
- ПИТАННЯ: це помилка чи design intent?
- АНАЛІЗ: В ICT/SMC доктрині:
  - CHoCH = підтвердження зміни напрямку (сильніший сигнал)
  - BOS = продовження тренду (слабший для trigger-а, але все ще structure event)
  - Для trigger "ready" — можливо тільки CHoCH мав би кваліфікувати
  - Але для "triggered"/"approaching" BOS цілком резонний
- ВАРІАНТИ:
  A) Перейменувати функцію на _find_qualifying_structure_break() (точніше)
  B) Фільтрувати тільки CHoCH для "ready", CHoCH+BOS для "triggered"
  C) Залишити як є (BOS теж підтверджує direction)
- РИЗИК: Якщо тільки CHoCH → менше "ready" станів → conservative але доктринально чистіше
- РЕКОМЕНДАЦІЯ: Варіант A (rename) — zero-risk cosmetic fix. Якщо є бажання доктринальної чистоти → B, але це вже P-slice.
- VERDICT: S3. Rename function → найпростіше. Detailed CHoCH/BOS split → окремий P-slice якщо потрібно.

#### ITEM-D: narrative config raw dict без typed parser (§14.2, §20 item 7) — S2 config
- ФАКТ: smc_runner.py читає cfg = config.get("smc", {}).get("narrative", {}) як raw dict
- Narrative config values доступаються через .get() з defaults у narrative.py
- РИЗИК: Якщо в config.json хтось напише "phase_hysteresis_bars": "3" (string) → порівняння int з string
  Реальний ризик низький бо config.json під контролем, не user input.
- ВАРІАНТИ:
  A) Створити SmcNarrativeConfig dataclass з from_dict() (як SmcDisplayConfig)
  B) Додати validation guard на вході synthesize_narrative()
  C) Залишити як є (config = SSOT, не user input)
- РЕКОМЕНДАЦІЯ: Варіант C зараз. Якщо буде runtime config reload або API config change → тоді A.
- VERDICT: S3 DEFERRED. Не production ризик. Записати як TECH DEBT.

#### ITEM-E: momentum_map dead input (§17) — S3 dead code

- РИЗИК: Zero production risk — dead code не ламає нічого, тільки зайве обчислення get_momentum_map()
- РЕКОМЕНДАЦІЯ: Варіант B (залишити як placeholder). Momentum → narrative integration логічна для v2.
  Але додати # TODO(momentum) коментар щоб не загубити intent.
